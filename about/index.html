全端代書控案系統 - 開發實錄
Phase 1: 需求探索 (Problem Discovery)
1. 現況 (The Context)

Scrivener Flow 是一個專為地政士（代書）設計的現代化案件管理 SaaS 平台，解決傳統代書作業依賴紙本、Line 群組與 Excel 追蹤的碎片化問題。

  • 系統架構： Next.js 14 (App Router) + Supabase (Auth, DB, Realtime)
  • 戰情儀表板： 整合「未來七日預告」與「稅單限繳監控」
  • 智慧解析： docx-parser 自動提取合約關鍵資訊
  • 自動化待辦： 基於里程碑自動推算任務
2. 痛點 (Pain Points)
  • 巨型函式與維護困難 (Monolithic Function)：

    原有文件解析邏輯長達 370+ 行，混雜檔案讀取與業務邏輯，難以測試與維護。

  • 資料狀態不同步 (Zombie Tasks)：

    缺乏「軟刪除」與同步機制，導致已刪除任務在重新整理後「死而復生」，嚴重影響信任度。

  • 高耦合與缺乏型別安全：

    早期開發缺乏嚴格型別 (any)，導致 Runtime 錯誤頻發。

3. 期待 (Expectations)
  • 架構解耦 (Decoupling)： 拆解複雜邏輯為純粹函式 (Pure Functions)。
  • 資料一致性 (Consistency)： 確保前後端狀態即時同步 (Single Source of Truth)。
  • 開發者體驗 (DX)： 引入 OpenSpec 標準化流程與 AI 協作。
  • 極致效能： 優化 Server Actions 與 Supabase 查詢 (Optimistic UI)。
Phase 2: 工程解法 (Engineering Solution)
6. 架構重構特寫 (Refactoring Journey)
從「毛線球」到「生產線」

早期的程式碼像一團糾纏的毛線（巨型函式），修改一個功能容易弄壞另一個。我們進行了大規模重構，將智慧合約解析變成一條標準化的工廠流水線：

原檔 Docx 格式轉換 平行提取 (日期/人名/金額) 整合驗證

效益： 新增合約格式的支援時間從 4小時 縮短至 30分鐘。

解決「殭屍任務」 (Zombie Tasks)

我們引入了智慧同步機制來解決已刪除任務自動復活的問題：

引入「軟刪除」與「版本比對」。系統會尊重您的刪除決定，只有在關鍵日期變更（如合約修改）時，才會「復活」重要任務提醒您。

4. 指定解法 (Specified Solutions)
A. 策略模式重構解析

將巨型函式拆解為 domain/case/types.ts 與多個獨立 Extractors。建立 Raw Docx -> Structured Data 的清晰 Pipeline。

B. 狀態管理與同步

引入 Soft Delete (is_deleted) 與智慧同步邏輯。前端採用 Optimistic UI，提升操作流暢度。

C. 導入工程化規範

OpenSpec (SDD) 規格驅動開發。嚴格執行 ESLint 與 CI/CD 自動化部署。

5. 驗收清單 (Acceptance Criteria)
功能

複雜合約解析 < 2秒

品質

無 any 型別
Zod 驗證 API

效能

儀表板 LCP < 1.5秒
零 Critical Errors

此專案展現了對「可維護性」、「資料一致性」與「用戶體驗」的極致追求。 親自體驗
